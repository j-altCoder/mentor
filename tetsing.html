01

Why Test At All?

Before writing a single line of test code, it's important to understand the problem testing solves. Skipping this leads to tests that feel like busywork.

Imagine you write a function calculateTotal(). It works great. Three months later, a teammate refactors a helper it depends on. Everything looks fine. But now calculateTotal() is silently returning wrong numbers — and nobody notices until a customer reports a billing error.

Tests are an automated safety net. They let you make changes with confidence, because if anything breaks, a test will catch it immediately — not a customer three weeks later.

Think of tests like a smoke detector in your house. You don't install one because you expect a fire. You install one so that if something ever goes wrong, you find out right away instead of discovering the damage days later.

What kinds of tests exist?

TypeWhat it testsSpeed
Unit testA single function or component in isolation. No network, no DB.Instant (~ms)
Integration testMultiple parts working together (e.g. a form + an API call).Fast (~seconds)
End-to-end (E2E)The full app in a real browser, simulating real user flows.Slow (~minutes)

Jest is primarily a unit and integration test runner. React Testing Library helps you write integration-style tests for React components. E2E testing (like Playwright or Cypress) is a separate topic entirely.

What is Jest exactly? Jest is a JavaScript test runner — a program that finds your test files, runs them, and tells you which passed and which failed. It also comes with utilities for making assertions, tracking function calls, and handling timers.
02

Setup & Config

There are two common setups: a plain Node.js project, and a React project. They use the same Jest core, but React needs extra tools to simulate a browser.

Plain Node.js Project

terminal
npm install --save-dev jest

Then in package.json, add a test script so you can run tests with npm test:

package.json
{
  "scripts": {
    "test": "jest",
    "test:watch": "jest --watch"   // re-runs on every file save
  }
}

React Project (Vite)

React components need a fake browser environment to render in — that's what jsdom provides. You also need Babel to understand JSX syntax.

terminal
# Core: Jest itself + fake browser (jsdom) + React renderer
npm install --save-dev jest jest-environment-jsdom
npm install --save-dev @testing-library/react @testing-library/jest-dom

# User interaction simulation
npm install --save-dev @testing-library/user-event

# Babel: lets Jest understand JSX and modern JS
npm install --save-dev babel-jest @babel/preset-env @babel/preset-react
jest.config.js
export default {
  // Use a fake browser (jsdom) instead of Node's bare environment
  testEnvironment: 'jsdom',

  // Run this file before every test — adds extra matchers like toBeInTheDocument()
  setupFilesAfterFramework: ['@testing-library/jest-dom'],

  // Tell Jest to run .js/.jsx files through Babel first
  transform: {
    '^.+\\.(js|jsx)$': 'babel-jest'
  },

  // CSS imports would crash Jest — this replaces them with a harmless mock
  moduleNameMapper: {
    '\\.css$': 'identity-obj-proxy'
  }
};
babel.config.cjs
module.exports = {
  presets: [
    ['@babel/preset-env', { targets: { node: 'current' } }],
    ['@babel/preset-react', { runtime: 'automatic' }]  // handles JSX
  ]
};
Using Create React App? CRA ships with Jest + RTL pre-configured. You can skip all of the above and just run npm test. The config is hidden inside the CRA package.
03

Your First Test

Let's build a test from scratch and understand every single piece of it.

Say you have a utility function in math.js:

math.js
export function add(a, b) {
  return a + b;
}

Now create a test file. Jest automatically finds files ending in .test.js or .spec.js, or anything inside a __tests__/ folder.

math.test.js
import { add } from './math';

// describe() groups related tests together under a label.
// Think of it like a folder for tests about the same thing.
describe('add()', () => {

  // test() (or it()) defines a single test case.
  // The first argument is a plain-English description of what you're checking.
  test('adds two positive numbers', () => {

    // Call your function with known inputs
    const result = add(2, 3);

    // expect() wraps a value so you can make assertions about it.
    // .toBe() checks that the value is exactly equal to 5.
    expect(result).toBe(5);
  });

  test('handles negative numbers', () => {
    expect(add(-1, -1)).toBe(-2);
  });

  test('returns 0 when both inputs are 0', () => {
    expect(add(0, 0)).toBe(0);
  });
});
The pattern is always: Arrange (set up inputs), Act (call the code), Assert (check the result). Experienced devs call this "AAA". Every test you ever write follows this same structure.

Running tests

npm test                  # run all tests once
npm run test:watch        # re-run whenever you save a file
npm test -- math          # run only files whose name contains "math"
npm test -- --verbose     # show every individual test name in output

What do test results look like?

✓ PASS  src/math.test.js
  add()
    ✓ adds two positive numbers (2ms)
    ✓ handles negative numbers
    ✓ returns 0 when both inputs are 0

✗ FAIL  src/math.test.js
  add()
    ✕ adds two positive numbers
      Expected: 5
      Received: 6    ← Jest tells you exactly what went wrong
What makes a good test name? Write test names as sentences describing the expected behavior: "returns an error when input is null" is far more useful than "test 1". When a test fails, the name is all you see in CI — make it informative.
04

Matchers

A matcher is the method you chain onto expect() to describe what you're checking. Jest has dozens. Here are the ones you'll use 90% of the time.

Equality

// toBe uses === (strict equality). Good for numbers, strings, booleans.
expect(2 + 2).toBe(4);
expect('hello').toBe('hello');

// toEqual does a DEEP comparison — checks every property recursively.
// Use this for objects and arrays.
expect({ name: 'Alice', age: 30 }).toEqual({ name: 'Alice', age: 30 });

// Why not just use toBe for objects?
expect({ a: 1 }).toBe({ a: 1 });  // ✗ FAILS — two different objects in memory
expect({ a: 1 }).toEqual({ a: 1 });  // ✓ PASSES — same structure

Truthiness

expect(true).toBeTruthy();     // passes for any truthy value: true, 1, "hi", []
expect(false).toBeFalsy();      // passes for: false, 0, "", null, undefined, NaN
expect(null).toBeNull();        // exactly null
expect(undefined).toBeUndefined();
expect('hello').toBeDefined();  // not undefined

Numbers

expect(10).toBeGreaterThan(5);
expect(3).toBeLessThanOrEqual(3);
expect(0.1 + 0.2).toBeCloseTo(0.3);  // use for floats — avoids 0.30000000000000004

Strings & Arrays

expect('Hello World').toMatch(/world/i);   // regex match
expect('Hello World').toContain('World');    // substring
expect([1, 2, 3]).toContain(2);             // array includes value
expect([1, 2, 3]).toHaveLength(3);          // array or string length

Errors

// IMPORTANT: wrap the call in an arrow function — don't call it directly.
// If you call it directly, the error throws BEFORE Jest can catch it.
expect(() => divide(1, 0)).toThrow();
expect(() => divide(1, 0)).toThrow('Division by zero'); // checks error message
expect(() => divide(1, 0)).toThrow(RangeError);        // checks error type

Negating with .not

// Put .not before any matcher to flip it
expect(5).not.toBe(10);
expect([]).not.toContain('foo');
expect(isAdmin).not.toBeTruthy();

Lifecycle Hooks

Sometimes every test in a file needs the same setup (open a DB connection, create a temp file) or the same cleanup (close connections, reset data). Lifecycle hooks let you do this without repeating yourself.

db.test.js
let db;
let userId;

// Runs ONE TIME before all tests in this file.
// Use for expensive setup that you only want to do once.
beforeAll(async () => {
  db = await Database.connect();
});

// Runs ONE TIME after all tests finish.
// Use for cleanup that mirrors beforeAll.
afterAll(async () => {
  await db.disconnect();
});

// Runs before EACH individual test.
// Use to put things back in a known state before every test.
beforeEach(async () => {
  userId = await db.insertUser({ name: 'Alice' });
});

// Runs after EACH individual test.
// Use to undo changes so tests don't bleed into each other.
afterEach(async () => {
  await db.deleteUser(userId);
});

test('reads user by id', async () => {
  const user = await db.findUser(userId);
  expect(user.name).toBe('Alice');
});

test('updates user name', async () => {
  await db.updateUser(userId, { name: 'Bob' });
  const user = await db.findUser(userId);
  expect(user.name).toBe('Bob');
});
Test Independence is Critical Tests must not depend on each other or share state. If test B only passes because test A ran first, you have a fragile suite. One test failure will cascade into many. Use beforeEach + afterEach to ensure every test starts clean.
06

Async Testing

Real apps do async things: fetch data from APIs, read files, query databases. By default, Jest won't wait for Promises to settle — you have to tell it to.

The Problem

// ❌ This test is broken — Jest doesn't wait for the promise to resolve.
// The test finishes instantly and PASSES even if fetchUser throws an error.
test('broken async test', () => {
  fetchUser(1).then(user => {
    expect(user.name).toBe('Alice');  // Jest doesn't reach this
  });
});

The Fix — async/await (recommended)

// ✅ Make the test function async and await the result.
// Jest will wait until the await resolves before checking assertions.
test('fetches user by id', async () => {
  const user = await fetchUser(1);
  expect(user.name).toBe('Alice');
});

Testing that a Promise rejects

test('throws 404 for unknown user', async () => {
  // await expect(...).rejects.toThrow() handles the rejection cleanly
  await expect(fetchUser(999)).rejects.toThrow('User not found');
});

Alternative — .resolves / .rejects matchers

// Slightly terser syntax. You MUST return (or await) the promise!
test('resolves with the right data', () => {
  return expect(fetchUser(1)).resolves.toMatchObject({ id: 1 });
});
Golden Rule for Async Tests Always await or return your async assertions. If you do neither, Jest considers the test passed the moment the synchronous code finishes — before any assertion even runs.

Mocking

Unit tests should be fast and isolated. But real code calls APIs, reads the filesystem, sends emails. Mocking replaces those real dependencies with fakes you control.

Imagine you're testing a function that emails a user a receipt. You don't want to actually send an email every time you run tests. A mock is like a dummy phone that records "someone tried to call this number" without actually dialing.

jest.fn() — Mock Functions

Creates a fake function that records every call made to it.

const sendEmail = jest.fn();  // does nothing by default, returns undefined

// If you need it to return something:
const greet = jest.fn(() => 'Hello!');

greet('Alice');
greet('Bob');

// Now you can ask Jest what happened
expect(greet).toHaveBeenCalled();
expect(greet).toHaveBeenCalledTimes(2);
expect(greet).toHaveBeenCalledWith('Alice');
expect(greet).toHaveBeenLastCalledWith('Bob');

Controlling return values

const fn = jest.fn();

fn.mockReturnValue(42);            // always returns 42
fn.mockResolvedValue({ id: 1 });     // returns a resolved Promise — use for async functions
fn.mockRejectedValue(new Error('fail')); // returns a rejected Promise
fn.mockImplementation((x) => x * 2); // full custom implementation

// Return different values on consecutive calls
fn.mockReturnValueOnce('first call')
  .mockReturnValueOnce('second call');

jest.mock() — Replace Whole Modules

When your code imports another module, you can tell Jest to replace that entire module with mocks before any test runs.

userService.test.js
// This line must be at the TOP, before imports (Jest hoists it automatically)
jest.mock('../api');

// Now fetchUser is a jest.fn() — a fake. The real network code never runs.
import { fetchUser } from '../api';
import { processUser } from '../userService';

test('processUser returns uppercase display name', async () => {
  // Tell the mock what to return for this test
  fetchUser.mockResolvedValue({ id: 1, name: 'Alice' });

  const result = await processUser(1);

  expect(result.displayName).toBe('ALICE');
  expect(fetchUser).toHaveBeenCalledWith(1);
});

jest.spyOn() — Watch Real Methods

A spy is like a mock but wraps an existing real function. The original code still runs, but you can also track calls or override it temporarily.

// Suppress console.error noise during tests
const spy = jest.spyOn(console, 'error').mockImplementation(() => {});

doSomethingThatLogsErrors();

expect(spy).toHaveBeenCalled();
spy.mockRestore();  // IMPORTANT: restore original behavior after the test

Cleaning up mocks between tests

afterEach(() => {
  jest.clearAllMocks();   // resets call history (how many times called, with what args)
  // jest.resetAllMocks()  — also removes return values / implementations
  // jest.restoreAllMocks() — also restores spyOn mocks to their originals
});
08

React Testing Library

Jest runs the tests. React Testing Library (RTL) gives you tools to render React components and interact with them — like a user would — inside those tests.

The Core Philosophy RTL's guiding principle: "Test your software the way users use it." That means don't query elements by CSS class names or component state. Find them the way a real user (or screen reader) would — by visible text, button labels, input names.

render() and screen

render() mounts your component into a fake DOM. screen is the object you use to search through that fake DOM.

Greeting.test.jsx
import { render, screen } from '@testing-library/react';
import '@testing-library/jest-dom';  // adds extra matchers like .toBeInTheDocument()
import Greeting from './Greeting';

test('displays the user name', () => {
  // 1. Render the component with props
  render(<Greeting name="Alice" />);

  // 2. Find the element. getByText finds by the text users see on screen.
  const heading = screen.getByText('Hello, Alice!');

  // 3. Assert about it. toBeInTheDocument() comes from @testing-library/jest-dom.
  expect(heading).toBeInTheDocument();
});
What does @testing-library/jest-dom add? Without it, you only have basic Jest matchers. The jest-dom package adds DOM-specific matchers like toBeInTheDocument(), toBeVisible(), toBeEnabled(), toHaveValue(), and more. It's almost always worth importing.
09

Querying the DOM

How you find elements in RTL is important. There's a recommended priority order, and then there's a choice between three variants (getBy, queryBy, findBy) depending on what you're doing.

Query Priority — use higher ones first

  • 1
    getByRole Finds by ARIA role + accessible name. Works for buttons, inputs, headings, links, dialogs, lists, etc. Best choice: matches how screen readers navigate the page. Forces you to write accessible markup.
  • 2
    getByLabelText Finds an input by its associated <label> text. Great for form inputs. Also encourages proper label usage for accessibility.
  • 3
    getByPlaceholderText Finds an input by its placeholder text. Use when there's no label. Less ideal — placeholders aren't great for accessibility.
  • 4
    getByText Finds by visible text content. Good for paragraphs, spans, list items. Not for interactive elements — use getByRole instead.
  • 5
    getByDisplayValue Finds an input/select/textarea by its current value. Useful when an input is already filled in.
  • 6
    getByAltText Finds elements (usually images) by their alt attribute. For images with descriptive alt text.
  • 7
    getByTestId Finds by data-testid attribute. Last resort only. If you're reaching for this, ask if the component is missing accessible markup. 
// getByRole examples — this is what you'll use most
screen.getByRole('button', { name: /submit/i });      // a button with text "Submit"
screen.getByRole('heading', { name: 'Welcome' });     // h1-h6 with text "Welcome"
screen.getByRole('textbox', { name: /email/i });     // input linked to "Email" label
screen.getByRole('checkbox', { name: /agree/i });    // checkbox with "Agree" label
screen.getByRole('link', { name: /home/i });         // anchor tag

// Using regex for name makes tests resilient to capitalization / whitespace changes
// { name: /submit/i } matches "Submit", "SUBMIT", "submit order", etc.

getBy vs queryBy vs findBy — which one to use?

getBy* — element must exist

Throws an error immediately if the element isn't found. Use when the element must be there right now. The error message is helpful for debugging.

screen.getByRole('button', { name: /save/i })

queryBy* — element might not exist

Returns null if not found instead of throwing. Use when asserting that something is absent.

expect(screen.queryByText(/error/i)).not.toBeInTheDocument()

findBy* — element will appear later

Returns a Promise. Retries automatically for up to 1 second until the element appears. Use for elements that appear after a fetch, animation, or state change.

const msg = await screen.findByText('Saved!')

*AllBy* — multiple elements

All three variants have an "All" version that returns an array. Use when you expect more than one match.

screen.getAllByRole('listitem') // returns array
Debugging: screen.debug() When a query fails and you're not sure why, call screen.debug() inside your test. It prints the current DOM to your terminal so you can see exactly what's rendered.

Simulating User Events

Finding elements is only half the battle. To test interactive components, you also need to simulate what users do: clicking buttons, typing in fields, selecting options.

RTL ships with a low-level fireEvent utility, but you should almost always use @testing-library/user-event instead. Here's why:

fireEvent.click(button) is like sending a single "click" signal directly to the DOM — it skips all the real browser behaviors that lead up to a click (focus, pointer down, pointer up, etc.).

user.click(button) fires all those events in the correct order, exactly as a real browser would. This catches bugs that only appear with real user interactions.
LoginForm.test.jsx
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import '@testing-library/jest-dom';
import LoginForm from './LoginForm';

test('submits with email and password', async () => {
  const handleSubmit = jest.fn();

  // Step 1: Set up userEvent — do this once at the top of each test
  const user = userEvent.setup();

  // Step 2: Render the component
  render(<LoginForm onSubmit={handleSubmit} />);

  // Step 3: Interact with it. user.type() clicks the field, then types each character.
  await user.type(
    screen.getByLabelText(/email/i),
    '[email protected]'
  );
  await user.type(
    screen.getByLabelText(/password/i),
    'mysecret123'
  );

  // Step 4: Click submit
  await user.click(screen.getByRole('button', { name: /log in/i }));

  // Step 5: Assert the result
  expect(handleSubmit).toHaveBeenCalledWith({
    email: '[email protected]',
    password: 'mysecret123'
  });
});

Common user actions

MethodWhat it simulates
user.click(el)Full mouse click on an element
user.dblClick(el)Double click
user.type(el, 'text')Click the field, then type character by character
user.clear(el)Select all text in field and delete it
user.keyboard('{Enter}')Press a specific key (doesn't focus — use after user.click)
user.tab()Press Tab to move focus to the next element
user.hover(el)Move mouse over an element (triggers :hover styles / tooltips)
user.selectOptions(el, ['b'])Select options in a <select> dropdown
user.upload(input, file)Attach a File object to a file input

Async UI Patterns

Components often fetch data, show loaders, then display results. Testing this requires waiting for the DOM to update — you can't just check immediately after render.

The scenario: a component that fetches on mount

// The component we're testing
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);

  useEffect(() => {
    fetch(`/api/users/${userId}`)
      .then(r => r.json())
      .then(setUser);
  }, [userId]);

  if (!user) return <p>Loading...</p>;
  return <h1>{user.name}</h1>;
}

Testing it with findBy

beforeEach(() => {
  // Mock the fetch call so we don't hit a real network
  jest.spyOn(global, 'fetch').mockResolvedValue({
    json: async () => ({ id: 1, name: 'Alice' })
  });
});

afterEach(() => jest.restoreAllMocks());

test('shows loading state then displays user name', async () => {
  render(<UserProfile userId={1} />);

  // Immediately after render, the fetch hasn't resolved yet — loading shows
  expect(screen.getByText('Loading...')).toBeInTheDocument();

  // findByRole polls the DOM until the heading appears (or 1 second passes)
  const heading = await screen.findByRole('heading', { name: 'Alice' });
  expect(heading).toBeInTheDocument();

  // Loading indicator should be gone now
  expect(screen.queryByText('Loading...')).not.toBeInTheDocument();
});

waitFor — when findBy isn't enough

findBy is shorthand for waitFor(() => getBy(...)). Sometimes you need to wait for something more complex — like multiple elements updating, or a mock being called.

import { waitFor } from '@testing-library/react';

// Wait until a condition is true (retries every 50ms for up to 1000ms)
await waitFor(() => {
  expect(mockSave).toHaveBeenCalledTimes(1);
  expect(screen.getByText('Saved!')).toBeInTheDocument();
});

// Wait for something to disappear
import { waitForElementToBeRemoved } from '@testing-library/react';

await waitForElementToBeRemoved(() => screen.getByText('Loading...'));
"not wrapped in act(...)" warning If you see this in your test output, it means a state update happened after your test finished — Jest noticed the DOM changed unexpectedly. The fix is usually to add an await findBy*(...) or await waitFor(...) to let the update finish before the test ends.

Testing Custom Hooks

React hooks can't be called outside a component — but renderHook from RTL gives you a lightweight "dummy component" wrapper so you can test hooks in isolation.

Say you have this hook:

// useCounter.js
export function useCounter(initialValue = 0) {
  const [count, setCount] = useState(initialValue);
  const increment = () => setCount(c => c + 1);
  const decrement = () => setCount(c => c - 1);
  const reset     = () => setCount(initialValue);
  return { count, increment, decrement, reset };
}
useCounter.test.js
import { renderHook, act } from '@testing-library/react';
import { useCounter } from './useCounter';

test('starts at the given initial value', () => {
  // renderHook renders the hook inside a throw-away component
  const { result } = renderHook(() => useCounter(5));

  // result.current holds whatever the hook returned
  expect(result.current.count).toBe(5);
});

test('increments the count', () => {
  const { result } = renderHook(() => useCounter());

  // Wrap state-changing calls in act() so React flushes the update
  act(() => result.current.increment());

  expect(result.current.count).toBe(1);
});

test('resets to initial value', () => {
  const { result } = renderHook(() => useCounter(10));

  act(() => result.current.increment());
  act(() => result.current.reset());

  expect(result.current.count).toBe(10);
});

If your hook needs a Context provider

// Pass a wrapper option to renderHook to supply any providers the hook needs
const wrapper = ({ children }) => (
  <AuthProvider user={{ id: 1, name: 'Alice' }}>
    {children}
  </AuthProvider>
);

const { result } = renderHook(() => useCurrentUser(), { wrapper });

Quick Reference Cheatsheet

Jest Matchers

.toBe(val)Strict equality (===). Primitives only.
.toEqual(val)Deep equality. Objects + arrays.
.toBeTruthy() / .toBeFalsy()Loose truthiness check.
.toBeNull() / .toBeUndefined()Exact null / undefined.
.toContain(item)Array has item OR string has substring.
.toHaveLength(n)Array or string length.
.toMatch(regex)String matches pattern.
.toThrow(msg?)Function throws error. Wrap in arrow fn!
.not.matcher()Negate any matcher.

jest-dom Matchers (RTL)

.toBeInTheDocument()Element exists in the fake DOM.
.toBeVisible()Not hidden (display, visibility, opacity).
.toBeEnabled() / .toBeDisabled()Input enabled/disabled state.
.toHaveValue('text')Input's current value.
.toHaveTextContent(/regex/)Element's text matches pattern.
.toHaveClass('active')Element has CSS class.
.toHaveAttribute('href', '/')Element has attribute + value.
.toBeChecked()Checkbox/radio is checked.
.toHaveFocus()Element has keyboard focus.

Mock Helpers

jest.fn()Create a mock function.
fn.mockReturnValue(v)Always return v.
fn.mockResolvedValue(v)Return resolved Promise(v).
fn.mockRejectedValue(e)Return rejected Promise(e).
fn.mockImplementation(fn)Full custom implementation.
fn.mockReturnValueOnce(v)Return v on the next call only.

The 5 Rules to Remember

1 — Test behavior, not implementationDon't test which state variables changed or what CSS classes were applied. Test what the user would see and do — visible text, button labels, form values.
2 — Use getByRole firstIt's the most realistic query. If you can't find your element by role, your component probably needs better accessible markup — which is a good problem to fix.
3 — Always await async assertionsIf you forget await or return on async matchers, the test passes vacuously. Jest does not warn you — it just silently skips the assertion.
4 — Each test must be independentUse beforeEach/afterEach to set up and tear down state. A test that only works if another test ran first is a bug in your test suite.
5 — Mock at the boundaryMock the thing your code depends on (an API, a module), not the internals of what you're testing. If you mock too deeply, you end up testing the mocks instead of your own code.
Part 2 · Practical MERN Testing

Building & Testing a MERN App

We build a Task Manager — a realistic mini-production app — and write tests at every layer: utility functions, Mongoose models, Express middleware, API routes, frontend fetch wrappers, React components, and custom hooks. Every test has a plain-English explanation of why it's written that way.

M-1

The App We're Building

Before writing a single test, understand the app you're testing. The structure below is what a real junior dev would encounter at their first job.

Users can register, log in, create tasks, mark them complete, and delete them. Here's the full project structure:

task-manager/
├── backend/
│   ├── models/
│   │   ├── User.js          # Mongoose schema — validation, virtuals, methods
│   │   └── Task.js          # Mongoose schema — isOverdue virtual

Testing Utility Functions

Always start here. Utilities are the simplest things to test — pure function in, pure value out. No database, no HTTP, no React. They run in milliseconds.

The utility code

backend/utils/password.js
const bcrypt = require('bcryptjs');

async function hashPassword(plain) {
  if (!plain || plain.length < 8) {
    throw new Error('Password must be at least 8 characters');
  }
  return bcrypt.hash(plain, 10);
}

async function verifyPassword(plain, hash) {
  return bcrypt.compare(plain, hash);
}

module.exports = { hashPassword, verifyPassword };
backend/utils/token.js
const jwt = require('jsonwebtoken');
const SECRET = process.env.JWT_SECRET || 'test-secret';

function signToken(payload) {
  return jwt.sign(payload, SECRET, { expiresIn: '7d' });
}

function verifyToken(token) {
  try {
    return jwt.verify(token, SECRET);
  } catch {
    return null;  // return null instead of throwing — callers check for null
  }
}

module.exports = { signToken, verifyToken };

Password tests

backend/utils/__tests__/password.test.js
const { hashPassword, verifyPassword } = require('../password');

describe('hashPassword()', () => {

  test('returns a bcrypt hash, not the plain password', async () => {
    const hash = await hashPassword('mypassword');
    expect(hash).not.toBe('mypassword');
    expect(hash).toMatch(/^\$2b\$/);  // all bcrypt hashes start with $2b$
  });

  test('produces a different hash each call — proves random salting', async () => {
    // If two hashes of the same password matched, attackers could use
    // pre-computed "rainbow tables" to reverse-engineer passwords.
    const [h1, h2] = await Promise.all([
      hashPassword('mypassword'),
      hashPassword('mypassword'),
    ]);
    expect(h1).not.toBe(h2);
  });

  test('throws when password is shorter than 8 characters', async () => {
    await expect(hashPassword('short')).rejects.toThrow('at least 8 characters');
  });

  test('throws when password is empty', async () => {
    await expect(hashPassword('')).rejects.toThrow();
  });
});

describe('verifyPassword()', () => {

  test('returns true when password matches the hash', async () => {
    const hash = await hashPassword('correctpassword');
    expect(await verifyPassword('correctpassword', hash)).toBe(true);
  });

  test('returns false when password is wrong', async () => {
    const hash = await hashPassword('correctpassword');
    expect(await verifyPassword('wrongpassword', hash)).toBe(false);

Testing Express Middleware

Middleware sits between the incoming request and your route handler. Testing it in isolation is fast and precise: you create fake req, res, and next objects, call the middleware directly, and check what it does — no HTTP server needed at all.

backend/middleware/auth.js
const { verifyToken } = require('../utils/token');

// Blocks any request that doesn't carry a valid JWT in the Authorization header
function requireAuth(req, res, next) {
  const authHeader = req.headers.authorization;

  if (!authHeader || !authHeader.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'No token provided' });
  }

  const decoded = verifyToken(authHeader.slice(7));  // strip "Bearer "
  if (!decoded) {
    return res.status(401).json({ error: 'Invalid or expired token' });
  }

  req.user = decoded;  // attach decoded payload for downstream handlers
  next();
}

module.exports = { requireAuth };
backend/middleware/validate.js
// Factory: returns middleware that checks the request body for required fields
function requireFields(...fields) {
  return function(req, res, next) {
    const missing = fields.filter(f => !req.body[f]);
    if (missing.length > 0) {
      return res.status(400).json({ error: `Missing fields: ${missing.join(', ')}` });
    }
    next();
  };
}

module.exports = { requireFields };

Testing API Routes with Supertest

Route (integration) tests fire real HTTP requests through your full Express stack — routing, middleware, controllers, and a real in-memory MongoDB — all in one. Supertest does this without binding to a port, so tests never conflict with each other.

npm install --save-dev supertest

    await request(app).post('/api/auth/register').send(validUser);
  });

      .post('/api/auth/login')

    expect(res.body.token).toBeDefined();

    const res = await request(app)
      .post('/api/auth/login')
      .send({ email: validUser.email, password: 'wrongpassword' });

Why "Invalid credentials" for both bad email AND bad password? If your API returns "User not found" vs "Wrong password" as separate messages, attackers can quietly discover which email addresses are registered just by trying logins. Returning the exact same message for both cases is a real production security requirement — and a perfect example of why you verify the exact error text in your tests.

Testing the Frontend API Layer

The frontend has a thin module wrapping all fetch calls. Testing it means mocking the global fetch function — no real network, no backend running. We verify that the right URLs, headers, and HTTP methods are sent, and that errors are handled correctly.

frontend/src/api/taskApi.js
const BASE = '/api';

function authHeaders(token) {
  return { 'Content-Type': 'application/json', 'Authorization': `Bearer ${token}` };
}

export async function fetchTasks(token) {
  const res = await fetch(`${BASE}/tasks`, { headers: authHeaders(token) });
  if (!res.ok) throw new Error('Failed to fetch tasks');
  return res.json();
}

export async function createTask(token, data) {
  const res = await fetch(`${BASE}/tasks`, {
    method: 'POST',
    headers: authHeaders(token),
    body: JSON.stringify(data),
  });
  if (!res.ok) throw new Error('Failed to create task');
  return res.json();
}

export async function deleteTask(token, id) {
  const res = await fetch(`${BASE}/tasks/${id}`, {
    method: 'DELETE',
    headers: authHeaders(token),
  });
  if (!res.ok) throw new Error('Failed to delete task');
}

import { useState } from 'react';

export function TaskForm({ onAdd, isLoading }) {
  const [title, setTitle] = useState('');

  function handleSubmit(e) {
    e.preventDefault();
    if (!title.trim()) return;  // ignore blank/whitespace-only input
    onAdd(title.trim());
    setTitle('');
  }

  return (
    <form onSubmit={handleSubmit}>
      <input
        type="text"
        value={title}
        onChange={e => setTitle(e.target.value)}
        placeholder="Add a new task..."
        aria-label="New task title"
      />
      <button type="submit" disabled={isLoading}>
        {isLoading ? 'Adding...' : 'Add Task'}
      </button>
    </form>
  );
}
frontend/src/components/TaskList.jsx
export function TaskList({ tasks, onDelete, onToggle }) {
  if (tasks.length === 0) return <p>No tasks yet.</p>;

  return (
    <ul aria-label="Task list">
      {tasks.map(task => (
        <li key={task._id}>
          <input
            type="checkbox"
            checked={task.completed}
            onChange={() => onToggle(task._id)}
            aria-label={`Mark "${task.title}" as ${task.completed ? 'incomplete' : 'complete'}`}
          />
          <span style={{ textDecoration: task.completed ? 'line-through' : 'none' }}>
            {task.title}
          </span>
          <button onClick={() => onDelete(task._id)} aria-label={`Delete "${task.title}"`}>
            Delete
          </button>
        </li>
      ))}
    </ul>
  );
}

Testing the useTasks Hook

The useTasks hook is the brain of the frontend — it loads tasks on mount, manages loading and error states, and exposes addTask and removeTask. We use renderHook to test state transitions directly, mocking the API module so no network calls happen.

frontend/src/hooks/useTasks.js
import { useState, useEffect, useCallback } from 'react';
import { fetchTasks, createTask, deleteTask } from '../api/taskApi';

export function useTasks(token) {
  const [tasks,   setTasks]   = useState([]);
  const [loading, setLoading] = useState(false);
  const [error,   setError]   = useState(null);

  useEffect(() => {
    if (!token) return;
    setLoading(true);
    fetchTasks(token)
      .then(setTasks)
      .catch(() => setError('Failed to load tasks'))
      .finally(() => setLoading(false));
  }, [token]);

  const addTask = useCallback(async (title) => {
    setLoading(true);
    try {
      const newTask = await createTask(token, { title });
      setTasks(prev => [...prev, newTask]);
    } catch {
      setError('Failed to add task');
    } finally {
      setLoading(false);
    }
  }, [token]);

  const removeTask = useCallback(async (id) => {
    try {
      await deleteTask(token, id);
      setTasks(prev => prev.filter(t => t._id !== id));
    } catch {
      setError('Failed to delete task');
    }
  }, [token]);

  return { tasks, loading, error, addTask, removeTask };
}
frontend/src/hooks/__tests__/useTasks.test.js
import { renderHook, act, waitFor } from '@testing-library/react';
import { useTasks } from '../useTasks';

// Replace the entire API module with jest.fn() mocks.
// The hook's job is to manage STATE — not to make real HTTP requests.
// Mocking the API boundary means we test the hook logic in isolation.
jest.mock('../../api/taskApi');
import { fetchTasks, createTask, deleteTask } from '../../api/taskApi';

const TOKEN = 'test-token';
const MOCK_TASKS = [
  { _id: '1', title: 'Buy milk',    completed: false },
  { _id: '2', title: 'Write tests', completed: true  },
];

beforeEach(() => jest.clearAllMocks());

test('starts loading, then populates tasks on success', async () => {
  fetchTasks.mockResolvedValue(MOCK_TASKS);
  const { result } = renderHook(() => useTasks(TOKEN));

  // Immediately after mount: loading is true, tasks is empty
  expect(result.current.loading).toBe(true);
  expect(result.current.tasks).toHaveLength(0);

  // Wait until the async fetchTasks resolves
  await waitFor(() => expect(result.current.loading).toBe(false));

  expect(result.current.tasks).toHaveLength(2);
  expect(result.current.error).toBeNull();
});

test('sets error state when fetchTasks rejects', async () => {
  fetchTasks.mockRejectedValue(new Error('Network error'));
  const { result } = renderHook(() => useTasks(TOKEN));

  await waitFor(() => expect(result.current.loading).toBe(false));

  expect(result.current.error).toBe('Failed to load tasks');
  expect(result.current.tasks).toHaveLength(0);
});

test('addTask() appends the returned task to the list', async () => {
  fetchTasks.mockResolvedValue(MOCK_TASKS);
  const newTask = { _id: '3', title: 'Call dentist', completed: false };
  createTask.mockResolvedValue(newTask);

  const { result } = renderHook(() => useTasks(TOKEN));
  await waitFor(() => expect(result.current.tasks).toHaveLength(2));

  // act() wraps state-changing calls outside of user events
  await act(async () => {
    await result.current.addTask('Call dentist');
  });

  expect(result.current.tasks).toHaveLength(3);
  expect(result.current.tasks[2].title).toBe('Call dentist');
  expect(createTask).toHaveBeenCalledWith(TOKEN, { title: 'Call dentist' });
});

test('removeTask() removes the correct task from state', async () => {
  fetchTasks.mockResolvedValue(MOCK_TASKS);
  deleteTask.mockResolvedValue();

  const { result } = renderHook(() => useTasks(TOKEN));
  await waitFor(() => expect(result.current.tasks).toHaveLength(2));

  await act(async () => {
    await result.current.removeTask('1');  // remove "Buy milk"
  });

  expect(result.current.tasks).toHaveLength(1);
  // "Buy milk" gone; "Write tests" remains
  expect(result.current.tasks[0].title).toBe('Write tests');
});

test('addTask() sets error state when createTask rejects', async () => {
  fetchTasks.mockResolvedValue([]);
  createTask.mockRejectedValue(new Error('Server down'));

  const { result } = renderHook(() => useTasks(TOKEN));
  await waitFor(() => expect(result.current.loading).toBe(false));

  await act(async () => {
    await result.current.addTask('New task');

  expect(result.current.error).toBe('Failed to add task');
  expect(result.current.tasks).toHaveLength(0);  // list unchanged on failure

Why mock the API module in hook tests? The hook's job is to manage state — not to make HTTP requests. By mocking taskApi.js we test exactly that: if fetchTasks resolves, does loading go false and tasks fill in? If it rejects, does error get set? We're testing the hook's brain, not the network.
M-9

Layer-by-Layer Summary

Here's a map of everything we tested and the core insight at each layer. When a test fails, this map tells you immediately where to look.

LayerToolWhat you're verifying
Utils (password, token) Jest only Pure function contract: correct output, input validation, edge cases (empty/tampered values)
Mongoose models Jest + mongodb-memory-server Schema validation, default values, lowercase transforms, unique indexes, virtuals, instance methods
Middleware Jest (no HTTP) next() called vs. 401 returned; req.user set correctly; all header format cases covered
API routes Jest + Supertest + mongodb-memory-server HTTP status codes, response shapes, auth guards, error messages, data isolation per user
Frontend API layer Jest (mock global.fetch) Correct URL, HTTP method, headers, JSON body; error thrown when response is not ok
React components RTL + userEvent Renders correctly, user interactions trigger right callbacks, loading/empty states display
Custom hooks RTL renderHook + jest.mock() loading/error/data state after mount; state after addTask/removeTask; error recovery
The single most important idea Each layer only tests its own responsibility. Utility tests don't care about HTTP. Route tests don't care about React. Component tests don't care about the database. When a test fails, you know exactly which layer broke — and that cuts debugging time from hours to minutes.
